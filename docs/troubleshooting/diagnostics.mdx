# 诊断方法

建议把问题处理固定成一套可重复执行的流程，这样可以降低支持成本，也能让博客和案例页复用这些方法。

## 第一步：确认范围

- 问题只影响单个服务，还是影响整个项目
- 问题只出现在某个地区、账号或时间窗口，还是普遍存在

## 第二步：采集信息

- 当前发布配置版本
- 目标服务地址与健康检查结果
- 请求时间、请求来源与错误日志

## 第三步：定位链路

从入口层、转发层、目标服务层依次排查，避免同时修改多个变量导致问题难以复现。

## 第四步：形成结论

记录根因、临时缓解方案、长期修复方案和是否需要补充文档或监控规则。
