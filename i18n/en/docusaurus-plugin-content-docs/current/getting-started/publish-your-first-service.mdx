# Create Your First Secure Access Path

Use a real workload for the first path validation so tenant isolation, ACL policy, relay behavior, and stability are all tested together.

## Suggested examples

- SSH access to build or debug hosts
- RDP access to remote desktop environments
- Web access to internal admin systems for operations teams

## Configuration flow

1. Select the destination device and confirm it belongs to the intended tenant.
2. Create or reuse tenant ACL rules for subject and time-window limits.
3. Select relay mode (shared, dedicated, or user-managed) for this path.
4. Initiate access and verify authorization plus connectivity.
5. Review session logs, ACL hit data, and stability signals.

## Validation checklist

- Unauthorized users are blocked by ACL
- Authorized users can access reliably across network conditions
- Relay unavailability triggers expected retry or fallback behavior
- Client runtime remains low-footprint and non-intrusive
- Audit records are complete and traceable
