# Product Architecture

The Larktun architecture can be read in four layers: the SaaS control site, the Headscale gateway and cluster, the relay network, and the client devices at each end. The embedded simulator on this page makes the key login, registration, direct path, and DERP relay flows much easier to understand.

## Architecture layers

- SaaS control site: handles sign-in, identity checks, access orchestration, and forwards control requests to the gateway.
- Headscale gateway and cluster: manage device registration, PreAuthKey delivery, node metadata, and secure network orchestration.
- Relay servers: carry traffic when P2P connectivity is unavailable and help maintain stability across regions or more complex network environments.
- Larktun clients: run on user laptops, managed desktops, or servers to establish encrypted paths and enforce access policy.

## Suggested walkthrough

1. Log in client A and client B one by one to observe how SaaS, the gateway, and Headscale participate in authentication and registration.
2. Adjust the Headscale and relay counts to see how the topology changes as the control plane expands.
3. Trigger direct networking to inspect how the clients establish a P2P data path.
4. Then trigger relay networking to compare how traffic falls back to DERP forwarding when direct connectivity is unavailable.

## What to observe

- The login path represents control-plane traffic rather than the final business data path.
- Direct connectivity stays the preferred route so traffic avoids relay when possible and latency remains lower.
- Relay is a resilience layer that helps in NAT-heavy, cross-region, or less predictable network conditions.
- Changing Headscale and relay counts is a useful way to explain scaling and high-availability choices in deployment guidance.
