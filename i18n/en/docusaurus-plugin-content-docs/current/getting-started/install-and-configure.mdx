# Deployment Modes and Configuration

Larktun deployment is usually a combination of tenant boundaries, ACL strategy, and relay model rather than one fixed topology.

## Recommended deployment models

### SaaS control plane

Best for most teams that need fast onboarding with managed tenant, policy, audit, and operations capabilities.

### SaaS + dedicated relay

Best for teams that require tighter latency and capacity control while keeping SaaS control-plane operations.

### SaaS + user-managed relay

Best for teams that need custom geography, network boundaries, or compliance placement.

## Planning checklist

- Tenant partitioning: separate by customer, business unit, or environment
- ACL model: define least-privilege access by role and device tags
- Relay placement: decide shared, dedicated, or user-managed with region layout
- Stability controls: reconnect behavior, health checks, and path monitoring
- Endpoint experience: ensure low footprint and non-intrusive runtime

## User-managed relay guidance

1. Start with one relay close to primary user regions.
2. Add health checks and alerting to maintain availability.
3. Add redundancy and failover drills for higher concurrency.
4. Apply minimal exposure and regular security patching.

## Configuration example

```yaml
deployment:
  controlPlane: saas
  tenantStrategy: by-customer
relay:
  mode: custom
  nodes:
    - name: cn-east-relay-1
      endpoint: relay-a.example.com:443
    - name: eu-central-relay-1
      endpoint: relay-b.example.com:443
acl:
  baseline: least-privilege
  approval: required-for-privileged-targets
stability:
  healthCheck: enabled
  autoFailover: enabled
agent:
  lowMemoryProfile: enabled
  silentMode: enabled
```

## Rollout guidance

- Template one tenant first, then replicate safely
- Standardize ACL and audit before scaling device count
- Validate failover for user-managed relay before production
